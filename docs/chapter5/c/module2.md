## 核心内容

### 一、Switch Transformer

Switch Transformer（Fedus et al., 2022）的核心目标：**在 FLOPs 不变的前提下极大扩展参数量**。

#### 1.1 Top-1 路由的极简哲学

```
为什么选 Top-1 而不是 Top-2?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Top-2 (如 Mixtral):                Top-1 (Switch):
  每个 token → 2 个专家               每个 token → 1 个专家

  优势: 表达能力更强                   优势:
  劣势: 通信量 ×2                       ① 路由计算量减半
         实现更复杂                       ② 每个 expert 的 batch 至少减半
         专家间需要加权融合                ③ 通信成本大幅降低
                                        ④ 实现极简

Switch 的关键洞察:
  "把 Top-2 的一份计算预算，拿去增加更多专家的参数量"
  与其让 2 个专家各算一半，不如让 1 个专家算满，用省下的通信开销
  放入更多专家 → 总参数量更大 → 模型容量更大
```

#### 1.2 容量控制与 Token 丢弃

```python
# Switch 的容量计算
capacity = (tokens_per_batch / num_experts) * capacity_factor

# capacity_factor 通常为 1.0~1.25
# capacity_factor = 1.0 时，每个专家恰好处理平均数量的 token
# 超出容量的 token 不执行 FFN，仅通过残差传递到下一层
```

#### 1.3 Router Z-loss：防止路由 logits 爆炸

$$\mathcal{L}_{total} = \mathcal{L}_{CE} + \lambda \cdot \log^2 \left(\sum_{i=1}^{N} \exp(z_i)\right)$$

```
Router Z-loss 为什么必要?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

问题: 路由器 logits = x · W_g
  → W_g 是 FP16/FP8 低精度
  → logits 值可能极大 → exp() 溢出 → softmax 爆炸 → 路由崩溃

Z-loss 的作用:
  惩罚归一化因子 Z = Σexp(z_i) 过大 → 约束 logits 幅度
  λ 通常很小 (如 0.001)，只做轻度约束
```

> 💡 **补充（PDF 课件 / CS336）**：Router Z-loss 的消融实验（750B tokens 训练）显示：**无 Z-loss 时训练 loss 出现剧烈尖峰**，HellaSwag/MMLU benchmark 波动巨大；有 Z-loss 后训练平滑、性能稳步提升。Router Z-loss 是 MoE 训练稳定的**必要组件**。

#### 1.4 小初始化

Switch Transformer 采用**截断正态分布**初始化（均值为 0，标准差适当减小），降低训练初期梯度方差，减少早期不稳定。

```
为什么小初始化对 MoE 特别重要?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

普通 Transformer:
  所有参数都参与每次计算 → 初始随机性被整体平滑

MoE:
  路由器随机初始化 → 早期路由几乎随机 → 梯度方向混乱
  如果初始化幅度大 → 梯度方差大 → 部分专家收到过大梯度 → 训练崩溃
  小初始化 → 早期变化温和 → 路由器有时间学到有意义的模式
```

#### 1.5 精度分级策略

```
精度分级设计:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

路由器 logits → FP32（关键决策，需要精度）
专家间传输  → FP16/bfloat16（平衡精度与带宽）
专家内部计算 → FP16/bfloat16（量大但可容忍）
All-to-All  → 可选 FP8（极致节省带宽）

原则: 关键路由决策用高精度，大量传输用低精度
```

> 📖 **延伸阅读**：各格式的原理、位布局、精度计算与选择依据，见 [数值理解](<数值理解.md>)。

#### 1.6 子层顺序的设计理由

Switch Transformer 采用 **Self-Attention → MoE/FFN** 的顺序，原因：

```
① Self-Attention: 建立全局语义 → 每个 token 融合上下文信息
② MoE/FFN: 基于丰富上下文的表征 → 路由器能更准确分配专家

如果反过来 (MoE → Attention):
  MoE 处理的是缺乏上下文的原始 embedding → 路由器判断不准
  非线性变换扰乱 token 几何关系 → 后续 Attention 相似度计算准确度下降
```

### 二、DeepSpeed-MoE：全栈优化

DeepSpeed-MoE 不做单一创新，而是从**模型结构→训练系统→推理加速**全栈优化。

#### 2.1 PR-MoE（参数效率）

```
PR-MoE 思路:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

标准 MoE:    每层 N 个专家，所有层相同
PR-MoE:     金字塔式专家分布

Layer 1-4:   2 个专家  ← 浅层需要的专业化较少
Layer 5-8:   4 个专家
Layer 9-16:  8 个专家  ← 深层需要更多专家精细分工

+ 固定 MLP（所有 token 共享）做"残差纠错"
  → 减少路由专家需要学习的通用知识
  → 参数效率提升
```

#### 2.2 MoS（蒸馏压缩）

```
PR-MoE（大） → MoS（小）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

两阶段蒸馏:
  Phase 1: 用教师信号训练学生 → 稳定学习教师分布
  Phase 2: 关闭蒸馏，只优化 LM loss → 学生发展自主泛化能力

效果: 模型尺寸再缩小 3.7×，推理速度比同等质量的稠密模型更快
```

#### 2.3 系统层优化

```
通信优化核心:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

① 自适应并行: 专家并行 + 专家切片 + 数据并行 + 张量切片灵活组合
   → 保证每层获得最合适的并行方式

② All-to-All 复杂度: 从 O(p) 降到 O(p/L)
   p = GPU 数量, L = 切片数

③ 分层 All-to-All:
   先节点内高速通信 → 再跨节点必要交换
   → 减少跨机器通信量

④ 稀疏重排 → 显式数据布局转换
   关键内核延迟下降 6×+
```

> 分层 All-to-All：把全局 All-to-All 拆成节点内 + 节点间两步执行，减少跨机器通信量。

### 三、MoE 与稠密模型对比

#### 3.1 核心区别

```
Dense Model:                     MoE Model:
━━━━━━━━━━━━━━━━━━               ━━━━━━━━━━━━━━━━━━
```

#### 3.2 MoE 的适用场景

```
✅ 适合 MoE:                      ⚠️ 不适合 MoE:
━━━━━━━━━━━━━━━━━━                ━━━━━━━━━━━━━━━━━━
大规模预训练                      小规模 / 资源受限
多 GPU / 多节点部署               单卡推理
数据量充足（> 数百B tokens）       数据量小
追求高模型容量                    追求简单稳定
```

### 四、Upcycling vs 从零训练

**Upcycling**：将已训练好的稠密模型改造为 MoE，复用已有权重。

**从零训练**：随机初始化 MoE，路由器和专家从头共同演化。

#### 4.1 两条路径的对比

```
Upcycling:
  稠密模型 FFN → 复制为 N 份 → 加入路由器 → 继续训练

  优势: 复用已训练成果，起步快
  风险: ① 稠密权重编码了强通用能力 → MoE 化后需"遗忘+重塑"
        ② 路由器随机初始化 → 早期分配模糊 → 路由器"学得太晚"

从零训练:
  随机初始化 → 路由器 + 专家共同进化

  优势: 专家从零分化，无历史包袱
  风险: 需要更多数据才能追上 upcycled 模型
```

#### 4.2 实证数据（存在矛盾！）

```
OLMoE 实验 (TC 路由):
━━━━━━━━━━━━━━━━━━━━━━━━
从零训练 MoE 在 ~500-600B tokens 时追上 upcycled 模型
≈ 原稠密模型训练数据量的 25%

Komatsuzaki Upcycling (EC 路由):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
从零训练需要 ~120% 的原训练量才能追上 upcycled 模型

差异原因: TC vs EC 路由策略 + 模型范式不同
```

#### 4.3 成功的 Upcycling 案例

> 💡 **补充（PDF 课件 / CS336）**：

**MiniCPM-MoE**（来源：CS336 课件）：
- 从 MiniCPM-2.4B 初始化，top-k=2, 8 experts
- MiniCPM-MoE(13.6B) 全面超越 MiniCPM-2.4B

### 📝 批改结果

**Q1 批改**：⚠️ 方向对但核心洞察缺失。

你提到"节省资源"和"知识稀疏分布"，方向没错，但**遗漏了 Switch 最关键的设计思想**：

Switch 的核心权衡不是"知识分布"，而是 **"计算/通信成本 vs 参数容量"** 的交换：
- Top-2：每个 token 经过 2 个专家 → 通信量 ×2、计算量 ×2、还需要加权融合两路输出
- Top-1：只走 1 个专家 → 上述成本全部减半

**省下来的资源做什么了？** 不是笼统的"通信和计算"，而是 **用来增加更多专家的数量**。这就是 Switch 的核心洞察：

> "与其让 2 个专家各算一半，不如让 1 个专家算满，用省下的开销放入更多专家 → 总参数量更大 → 模型容量更大"

例如：同样的 FLOPs 预算，Top-2 用 4 个专家，Top-1 可以用 8 个专家（参数量翻倍）。

— 得分：**5/10**

**Q2 批改**：⚠️ 相似点答对了，但两个关键点有误。

✅ 正确的部分：
- "都是引入惩罚项防止极端值" → 对
- "router z-loss 惩罚 softmax 的分母" → 准确，log²(Z) 中的 Z 就是归一化因子

❌ 两个问题：

① **混淆了 Router Z-loss 和 Auxiliary Loss 的作用**

你说的"某个 expert 经常被 route 到，其他专家路由少，训练不均衡"——这是 **Auxiliary Loss（负载均衡损失）** 的职责，不是 Z-loss 的。

Router Z-loss 的核心目的是 **数值稳定性**（防止低精度下 exp() 溢出），不是负载均衡。我们在对话中详细讨论过这个话题——Z-loss 直接约束的是 log(Z)，防止 softmax 计算时归一化因子过大导致溢出或下溢。

② **稠密模型不需要的原因不完整**

"稠密模型没有 router"是对的，但更深一层：稠密模型的输出层 softmax 也有类似问题（logits 过大 → 数值不稳定），所以 z-loss 在稠密模型中也存在，只是作用在**输出层**而非路由层。MoE 特殊在于**每一层都有路由 softmax**，低精度下的数值问题更频繁、更危险。

— 得分：**5/10**

**Q3 批改**：⚠️ 基本直觉正确，但分析不够深入。

✅ 你抓住了两个关键点：
- TC 保证所有 token 都参与（残差传递兜底）
- EC 可能导致某些 token 被多个专家重复选择，另一些 token 完全没被选中

❌ 缺少对 **upcycling 与从零训练差距** 的分析：

问题的核心不是"TC vs EC 谁训练更高效"，而是 **"从零训练追上 upcycled 模型的速度为什么不同"**。

关键机制：
- **TC + Upcycling**：专家已有好权重，但路由器是随机的 → 路由器需要时间学会正确分配 → 起步有优势但后期提升慢（路由器和专家不匹配）
- **TC + 从零**：路由器和专家从零共同进化 → 早期都差，但**没有历史包袱** → 后期加速追上
- **EC + Upcycling**：专家主动选择 token，好专家立刻能做出好的选择 → 起步优势巨大
- **EC + 从零**：随机专家做选择 → 选择质量极差 → 训练效率低 → 需要极长时间才能追上

所以 TC 下从零训练更容易追上的根本原因是：**TC 的 upcycled 优势相对较小**（路由器需要重新学习），而 EC 的 upcycled 优势巨大（好专家直接做好选择）。差距小 → 追上快。

— 得分：**6/10**
