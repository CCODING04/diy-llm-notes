**Q**：我看到了教程中"MoE 的实际效果"对比：DeepSeek 的实验 Pile Loss 可以降低更多（Pile Loss 是什么？可以解释下吗）GSM8K 上 MoE 的性能甚至超越了 Dense。以上现象的原因是什么？（明明激活的参数量更少）

**A**：

**Pile Loss 是什么？**

Pile 是一个大规模文本数据集（由 EleutherAI 创建，约 825GB 文本），常用于 LLM 预训练评估。Pile Loss 就是模型在 Pile 数据集上的交叉熵损失（Cross-Entropy Loss）——即模型预测下一个 token 的平均负对数似然：

$$\text{Pile Loss} = -\frac{1}{N}\sum_{i=1}^{N} \log P(x_i | x_1, ..., x_{i-1})$$

它是衡量语言模型预训练质量的核心指标：Loss 越低，模型对语言的建模能力越强。不同模型用同一个数据集评估，Loss 可直接比较（控制了数据变量）。

**为什么 MoE 激活参数更少却更优？**

关键在于区分 FLOPs（实际计算量）和参数容量（总参数量）：

```
Dense 0.2B:    参数 0.2B    →  每次计算使用 0.2B   →  FLOPs = X
MoE 2.0B:      参数 2.0B    →  每次计算使用 0.2B   →  FLOPs = X（相同！）
                                              ↑ 只激活 k 个专家
```

两者 FLOPs 相同，但 MoE 的 2.0B 参数提供了远大于 0.2B 的"知识库"，路由器从中挑选最相关的 0.2B 来回答——类比：Dense 像一个通才，0.2B 参数什么都要学，能力有限；MoE 像一个拥有 10 个专才的团队（共 2.0B 知识），每次只请最合适的专才回答（0.2B 计算量），但可调用的知识总量是 10 倍。

GSM8K 反超的原因更具体：GSM8K 是数学推理任务，需要特定类型的能力（计算、逻辑）。MoE 的路由器可以学会将数学相关 token 路由到"数学专家"。该专家的 0.2B 参数全部用于数学能力，而 Dense 7B 的数学相关参数散布在通用权重中，不如专家集中。这就是 MoE 的专家专业化优势——激活的参数虽然少，但针对当前任务的质量更高。

**Q**：关于噪声门控（Noisy Gating）部分，加入噪声是为了让训练过程中数据只路由到既定的几个专家，让所有专家都能从知识中获取内容，但是我有几点疑问：
1. softplus 函数来源于什么，如何实现，为什么选择 softplus
2. W_noise 是可学习变量，在训练过程中引入，并且加入了标准正态分布作为因子来控制变化，这部分在推理时会去掉，是否对推理性能有影响 -> 毕竟引入了随机性相当于对专家的内容进行了污染，可能不再 focus
3. 是否有方案让 x·W_g 更聚焦（经过 softmax 后更窄，或者有比较窄的 k 个值）；或者有办法将 gate 在早期更随机，在训练中期或者晚期更聚焦？这样是否会有助于 MoE 在早期每个专家能见识更多知识，在训练中晚期产生分化，更聚焦

**A**：

**1. Softplus 函数**

来源与定义：Softplus(x) = ln(1 + eˣ)，是 ReLU 的平滑近似。

```
函数形状对比:

ReLU:    max(0, x)         ← 折线，x<0 时为 0，x>0 时为 x
Softplus: ln(1 + eˣ)       ← 平滑曲线，处处可微
Sigmoid:  1 / (1 + e⁻ˣ)    ← S 形曲线，值域 (0, 1)

值域对比:
  Sigmoid:   (0, 1)       ← 有上界，最大噪声系数 = 1
  Softplus:  (0, +∞)      ← 无上界，可以产生很大的噪声
```

PyTorch 实现：torch.nn.functional.softplus(x) 或手动 torch.log(1 + torch.exp(x))

为什么选 Softplus 而不是 Sigmoid？关键在于值域：
- Sigmoid 映射到 (0, 1) → 噪声标准差最大为 1，限制了探索幅度
- Softplus 映射到 (0, +∞) → 模型可以学到让某些 token 的噪声很大（强探索），某些很小（高确定性），给模型更大的自由度

```
W_noise 的输出 → Softplus → 噪声标准差 σ

训练早期: W_noise 随机初始化 → σ 较大 → 噪声较强 → 所有专家都有机会
训练后期: W_noise 学到分化 → 对"确定"的 token σ≈0，对"模糊"的 token σ 较大
                              ↑ Softplus 允许 σ→0（几乎无噪声）
                              ↑ Sigmoid 下限约 0.5，无法完全关闭噪声
```

Mini LLM+MoE.py 代码中用 Sigmoid 是简化版本，原始 Shazeer 论文用 Softplus 更合理。

**2. 推理时去掉噪声是否有影响？**

去掉噪声对推理性能没有负面影响，反而是最优选择。

噪声的作用不是"污染专家"，而是"保证专家都能被训练到"：

```
训练时（有噪声）:
Token "猫" → 路由器原始选择 Expert 3（得分最高）
           → 加入噪声后可能变成 Expert 7（噪声扰动）

效果: Expert 7 本来永远接收不到"猫"相关 token
     → 噪声强迫它处理 → Expert 7 学到关于"猫"的知识
     → 所有专家都得到训练信号，不会"饿死"

推理时（无噪声）:
Token "猫" → 路由器确定性选择 Expert 3（最匹配）

效果: 每个专家已经在训练阶段学到了足够多样的知识
     → 推理时选最匹配的专家就行，不需要再探索
```

类比：噪声就像学生时代的"通识教育"（强制学各种科目），推理时去掉噪声就像工作后的"专业对口"（只做最擅长的事）。通识教育让学生的基础更扎实，不会影响专业表现。

如果推理时保留噪声会怎样？模型输出会变成随机波动的——同一个输入每次得到不同结果。这不是多样性，而是退化（模型已经学会了确定性路由，强行加噪声只会引入噪声）。唯一可能的用途是集成推理（多次采样取平均），但用更大的 k 通常更高效。

**3. 早期探索、晚期聚焦的策略**

你的直觉完全正确，这正是 MoE 研究的前沿方向。现有方案：

**方案 A：温度退火（Temperature Annealing）**

gate_scores = Softmax(logits / τ)
训练早期: τ = 2.0 (高温) → Softmax 更平坦 → 分布更均匀 → 专家负载均衡
训练中期: τ = 1.0 (正常)
训练后期: τ = 0.5 (低温) → Softmax 更尖锐 → 更聚焦于最优专家

**方案 B：噪声幅度衰减**

H(x)_i = (x·W_g)_i + α(t) · StandardNormal() · Softplus(x·W_noise)
α(t) = max(0, 1 - t / T_noise)
T_noise 之前: α 从 1 衰减到 0（探索→聚焦）
T_noise 之后: α = 0（纯确定性路由）

**方案 C：DeepSeek-V3 的 Bias-based 均衡**

不依赖噪声，而是给每个专家维护一个可调节的 bias：
logits_adjusted = x·W_g + bias[e]
训练中动态更新 bias：Expert e 处理 token 太多 → 减小 bias[e]；处理太少 → 增大 bias[e]
特点：早期 bias 接近 0→路由由数据决定；训练中动态调整→自动均衡；不引入噪声→不"污染"专家训练；无辅助损失→不干扰主训练目标

**方案 D：三阶段训练策略（CS336 课件提到的思路）**

Phase 1: 混沌初开（Warm-up）- 路由器自由探索，共享专家学通用知识
Phase 2: 职业规划 - 暂停训练，分析专家激活模式，识别敏感专家
Phase 3: 定向深造 - 共享专家继续学通用能力，路由专家仅在关键输入上激活

各方案对比：

| 方案 | 探索→聚焦机制 | 额外参数/损失 | 实际采用 |
|------|:-----------:|:----------:|:-------:|
| 温度退火 | τ 递减 | 无额外参数 | 研究阶段 |
| 噪声衰减 | α(t) 递减 | W_noise | Shazeer 2017 |
| Bias-based | 动态 bias | bias 向量 | DeepSeek-V3 |
| 三阶段 | 手动切换 | 需要暂停训练 | 理论框架 |

你的直觉（早期探索、晚期分化）与这些前沿方案完全一致。核心矛盾就是：早期需要均匀分配让所有专家都能学到东西，晚期需要聚焦让每个专家发展专长。噪声门控是最简单的实现方式，DeepSeek-V3 的 bias-based 方案是更优雅的解法。

**Q**："在批处理推理中，其他用户的查询可能导致你的 token 被丢弃"其他用户的查询是指什么行为？

**A**：

这是指生产环境中的 API 批处理推理。用一个具体例子解释：

假设你部署了一个 MoE 模型的 API，同时有多个用户在调用：

```
GPU 0 上同时处理的请求:

用户 A: "请帮我翻译这段话..."     → 生成中，20 个 token
用户 B: "今天天气怎么样？"         → 生成中，8 个 token
用户 C: "写一个排序算法..."        → 生成中，15 个 token

批处理: 把三个用户的 token 拼成一个大 batch
       → 总共 43 个 token 一起送入 MoE 层
```

容量限制是按整个 batch 计算的：

```
MoE 层处理:
  N_total = 43 (整个 batch 的 token 总数)
  num_experts = 4
  capacity_factor = 1.25
  每个专家容量 = (43 / 4) × 1.25 ≈ 13 个 token
```

丢弃如何发生：

```
路由结果（假设）:

Expert 0 被选中的 token:
  用户 A: token 1, 3, 7, 12, 15    (5 个)
  用户 B: token 22, 25, 27          (3 个)
  用户 C: token 31, 33, 35, 37, 38, 40, 42  (7 个)
  总共: 15 个 > 容量 13 个!

前 13 个被处理（假设按顺序）:
  ✅ 用户 A: 全部处理 (5 个)
  ✅ 用户 B: 全部处理 (3 个)
  ❌ 用户 C: token 40, 42 被丢弃! (超出容量)
              ↑ 这两个 token 仅通过残差传到下一层
                未经专家处理 → 语义信息缺失
```

用户 C 的 token 被丢弃，不是因为用户 C 自己的问题，而是因为用户 A 和 B 的 token 碰巧也选中了同一个专家，先把容量占满了。

根本原因是 MoE 的容量限制是全局批处理级别的，不是按用户隔离的。

工程上的应对：
- 增大 capacity_factor（如 1.5 或 2.0）→ 减少丢弃概率，但浪费计算
- 动态批处理：根据路由预测调整 batch 组成，减少专家冲突
- 用户间隔离：某些服务对 VIP 用户独占 batch，避免被"挤掉"
- 推理时不丢弃：推理阶段通常设 capacity_factor 足够大或不做容量限制，丢弃主要发生在训练阶段控制计算量

**Q**：MoE 模型是普遍发生这种现象吗？那理解 MoE 模型在高负载下的"降智"行为，是不是可以理解为某个用户 token 数量较多，并且刚好处在最后一批（或者用户优先级比较低，付费等级较低等原因给放到最后一批），那么该用户的 token 有可能不全部经过 expert，而直接被丢到下一层。又因为这种"现象"是在输入阶段确定的，所以整个模型的所有专家层，这个用户的 capacity 外的 token 都不会经过专家层，因此导致输出结果不理想或者"降智"？

**A**：

你的推理链条有一部分是对的，但有一个关键误解需要纠正。

**纠正：不是"同一批 token 在每层都被丢弃"**

每个 MoE 层的路由决策是独立的——每层有自己的路由器（独立的 W_g），token 的隐藏状态在每层之后都会变化：

```
同一条 token "猫" 在不同层的路由命运:

Layer 2 (MoE): "猫" → Expert 3 (容量已满) → ❌ 被丢弃 → 只走残差
Layer 4 (MoE): "猫" → Expert 1 (容量充足) → ✅ 正常处理
Layer 6 (MoE): "猫" → Expert 5 (容量充足) → ✅ 正常处理
Layer 8 (MoE): "猫" → Expert 3 (这次没满) → ✅ 正常处理

每层的路由器不同，token 的隐藏表示也在变 → 路由选择不同
```

所以不是"从头到尾某条 token 都不经过专家"，而是在某些层碰巧被丢弃、某些层正常处理。

**但你的核心直觉是对的——存在"级联退化"**

虽然不是每层都丢，但早期层的丢弃会导致后续层的隐藏状态"营养不良"：

```
正常路径:
  x → Layer2(Expert处理) → x' (经过专家非线性变换，语义丰富)
                         → Layer4(Expert处理) → x'' → ...

被丢弃路径:
  x → Layer2(仅残差, x_out = x) → x' (= x, 没有任何变换！)
                                  → Layer4(Expert处理) → x'' → ...
                                   ↑ 输入的隐藏状态"营养不良"
                                     Layer4 处理的是一个未经充分变换的 x
                                     输出质量必然受影响
```

级联效应：假设模型有 4 个 MoE 层，某 token 在 Layer2 被丢弃 → Layer2 丢弃 ✗ 隐藏状态弱于正常 → Layer3 Attention 正常 ✓ 但输入已"受损"，注意力效果打折 → Layer4 MoE 正常 ✓ 但输入不够好，输出也打折 → 最终退化是"渗入式"的，不是"断层式"的。

这是普遍现象吗？在生产推理中，token 丢弃其实不常见。训练阶段 capacity_factor = 1.0~1.25 有意丢弃以控制计算量；推理阶段 capacity_factor = 很大或 ∞ 尽量不丢弃，保证输出质量。

那高负载下的"降智"到底是什么原因？MoE 模型在高负载下的退化，token 丢弃只是理论可能，更常见的原因是：

| 退化原因 | 机制 | 影响 |
|---------|------|------|
| KV Cache 淘汰 | 高并发时显存不够，强制截断历史 context | 丢失长程依赖，"忘记"之前的对话 |
| 量化精度下降 | 高负载时为节省显存切换到更低精度（FP16→INT8→INT4） | 细粒度语义丢失 |
| Batch 截断 | 长序列被截短以适配 batch | 用户长输入被截断 |
| 动态批处理延迟 | 为凑 batch 等待导致超时或降级服务 | 低优先级用户响应慢或被拒 |

总结：
- 需要修正：~~同一 token 在所有 MoE 层都被丢弃~~ → 每层独立路由，命运不同
- 需要修正：~~降智 = token 永远不经过专家~~ → 降智 = 某些层的丢失导致隐藏状态逐层累积退化
- 正确：长序列用户 / 低优先级用户确实更容易受影响
- 正确：输入阶段的路由决定会影响后续所有层的处理质量
- 正确：MoE 模型在高负载下确实存在服务质量不稳定的工程挑战

**Q**：LSH（局部敏感哈希）中，epsilon 控制桶的宽度，是如何做到的？（感觉只是等比例放缩，没有其他操作，相当于对分布进行了压缩吗？）

**A**：

你的直觉"等比例放缩"基本正确，但关键是理解放缩 + floor 函数的组合效果。

拆解公式 $h_i(x) = \lfloor (a_i^\top x + b_i) / \epsilon \rfloor$：

分三步看：
① a_i^T · x: 随机投影，将 d 维向量投影到 1 维实数轴上
② + b_i: 随机偏移，打破对齐（避免边界效应）
③ / ε → floor: 这才是核心——量化分桶

ε 的作用：控制 floor 的"网格宽度"。用具体数字演示，假设 6 个 token 的投影值为 [0.3, 0.7, 1.2, 1.8, 2.5, 3.1]：

```
ε = 1.0 (窄桶):
除以 ε: [0.3, 0.7, 1.2, 1.8, 2.5, 3.1]
floor:  [ 0,   0,   1,   1,   2,   3 ]  → 4 个桶
桶宽 = ε = 1.0，每个桶在原始空间覆盖 [0,1), [1,2), [2,3), [3,4)

ε = 2.0 (宽桶):
除以 ε: [0.15, 0.35, 0.6, 0.9, 1.25, 1.55]
floor:  [  0,    0,    0,   0,    1,    1 ]  → 2 个桶
桶宽 = ε = 2.0，每个桶在原始空间覆盖 [0,2), [2,4)
→ 原来 4 个桶合并为 2 个，每个桶装更多 token

ε = 0.5 (极窄桶):
除以 ε: [0.6, 1.4, 2.4, 3.6, 5.0, 6.2]
floor:  [ 0,   1,   2,   3,   5,   6 ]  → 6 个桶
桶宽 = ε = 0.5，分得更细，每个桶装更少 token
```

所以你说的"压缩分布"对了一半。ε 确实在放缩，但效果不只是"压缩"：

① 桶宽度: 每个桶在原始投影空间覆盖的宽度 = ε
   ε 小 → 桶窄 → 精细分桶 → token 分散到更多桶
   ε 大 → 桶宽 → 粗略分桶 → token 集中到更少桶

② 相似性阈值: 两个 token 的投影差 < ε → 一定在同一桶（或相邻桶）
   ε 小 → 只有非常相似的 token 才会分到一起（高精度，低召回）
   ε 大 → 不太相似的 token 也会分到一起（低精度，高召回）

本质：ε 不是在压缩分布的形状，而是在调节量化的分辨率。类比相机像素：ε 小 = 高分辨率（每个像素覆盖的面积小，细节多）；ε 大 = 低分辨率（每个像素覆盖的面积大，细节少但画面紧凑）。

最后取模映射到专家：h 的范围取决于 ε。ε = 0.1 → h 可能是 0~100+ → % num_experts 分布较均匀；ε = 10 → h 可能是 0~3 → % num_experts 分布较粗糙。ε 通过控制 h 的离散度，间接影响取模后专家分配的均匀性。

**Q**：这种数学相关的思想映射到现实的处理方案中，是怎么思考出来的，应该如何培养这种思维，或者看哪些书做哪些课程能培养这种思维？（感觉自己很多年的数学思维白学了，完全想不到这种设计，或者看公式没有联想到"桶"的处理方式）

**A**：

先说一个残酷的事实：这不是"数学思维"。

你能看懂公式 $h(x) = \lfloor (ax+b)/\epsilon \rfloor$ 的每一步运算，这是计算能力。但看到 floor 就想到"分桶"，看到取模就想到"映射到专家"——这不是数学，这是工程直觉。

```
纯数学思维:                    工程直觉:
"floor 是取整函数"             "floor 是把连续量切成离散格子"
"除以 ε 是标量乘法"           "除以 ε 是调节分辨率/精度"
"取模是求余数"                 "取模是哈希映射/负载分配"
"这个函数的性质是..."          "这个函数能解决什么实际问题？"
```

这两种思维几乎没有交叉训练。学校数学课培养的是前者，后者来自另一个领域。

这种直觉从哪里来？LSH 的 floor 分桶不是某天才凭空想到的，而是从信号处理借来的：

```
思想源头链:

信号处理 (1940s-):
  连续模拟信号 → 采样 → 量化(quantization) → 数字信号
  量化的核心: floor(value / step_size) → 离散编码
  ↑ 这就是 ε 分桶的原型

↓ 思想迁移

信息检索 (1990s):
  高维向量 → 随机投影 → 量化分桶 → 近似最近邻查找
  Indyk & Motwani (1998) 提出 LSH

↓ 再次迁移

MoE 路由 (2020s):
  Token 嵌入 → 随机投影 → 量化分桶 → 取模映射到专家
  ↑ 原封不动地复用了信号处理的量化思想
```

核心发现：这个思维路径不是"数学好就能想到"，而是"见过信号处理/信息检索中的类似方案，迁移过来的"。

如何培养这种直觉？不是"学更多数学"，而是学那些"把数学变成系统"的领域：

第一梯队：直接相关的课程/书

最实用的一个练习：下次看到一个公式时，不要问"它是什么"，而是问三个问题：
1. 这个运算在物理世界中对应什么？（floor → 切割/分格/量化；/ε → 调节精度/分辨率；%N → 分配到 N 个槽位）

## 模块 2：MoE 变体与稠密模型对比 — QA 记录
> 📅 2026-04-23 ~ 2026-04-24

**Q**：关于 Router Z-loss 的公式：$$\mathcal{L}_{total} = \mathcal{L}_{CE} + \lambda \cdot \log^2 \left(\sum_{i=1}^{N} \exp(z_i)\right)$$ 其中 Z-loss 的设计原理是什么，为什么不能直接用正则抑制呢？

**A**：

核心问题是：**softmax 的数值稳定性取决于归一化因子 Z = Σexp(z_i)**，而不是单个 z_i 的大小。

Z-loss 惩罚的是 `log²(Z)`——即**归一化因子的对数的平方**。它约束的恰好是 softmax 计算中会溢出的那个量。

为什么普通正则不行？

```
方案 1：Weight Decay（对 W_g 做 L2）
惩罚: ||W_g||²
问题: z = x · W_g，即使 W_g 很小，如果 x 很大 → z 依然很大
     → 间接约束，不保证 log(Z) 小
     → 还会削弱路由器的区分能力

方案 2：L2 正则（直接对 z_i）
惩罚: Σ z_i²
问题: ① 不精确——我们要控制的是 Z = Σexp(z_i)，不是 z_i 的平方和
     ② 会惩罚所有大的 logit，包括那些"本就应该大"的（正确路由的专家）
     ③ 干扰路由器的判别力——你希望 logits 之间有差异，只不想它们整体过大

方案 4：Z-loss ✅
惩罚: log²(Z) = log²(Σexp(z_i))
优势: ① 精准——直接约束 softmax 计算中溢出的那个量
     ② 等比例缩放不变——如果所有 z_i 减去同一个常数 c：
        softmax 不变（因为分子分母的 exp(c) 约掉）
        但 L2(z) 会改变——说明 L2 在约束一个不影响 softmax 的量
     ③ 只约束"整体激活能量"，不限制 logits 的相对差异
```

Z-loss 的核心作用：只约束"温度"（logits 整体尺度），不约束"方向"（logits 间的相对排序）。

```
logits = [2.0, 1.0, 0.5]     → softmax ≈ [0.59, 0.22, 0.13]  ← 正常路由
logits = [200, 100, 50]      → softmax ≈ [1.0, 0.0, 0.0]      ← 方向一样但极端

两种情况下 argmax 结果相同，路由选择一样。
区别只是"温度"不同——前者温和，后者极端。

Z-loss 的作用：
  惩罚 log(Z) 过大 → 约束 logits 整体尺度 → 防止温度过低（过度自信）
  同时不影响 logits 之间的相对排序 → 路由决策不变
```

为什么是 `log²(Z)` 而不是 `log(Z)`？

一次方 log(Z): 惩罚力线性增长 → 对很大的 Z 约束力不够
二次方 log²(Z): 惩罚力超线性增长 → Z 越大，推回的力越强 → 在 log(Z) 附近建了一道"软墙"

这也就是 λ 通常设得很小（0.001）的原因——惩罚增长太快，λ 大了反而会过度压制路由器的表达能力。

**Q**：上面解释得非常好了！但是中间关于 FP16 溢出，"logits = [200, 100, 50] → softmax ≈ [1.0, 0.0, 0.0] ← FP16 溢出！但方向一样"，这里面哪里会有溢出，FP16 的范围应该是[-65504, 65504]，因此应该不存在溢出才对。

**A**：

你说得对，logits 本身不会溢出 FP16，溢出发生在 **exp()** 这一步。

```
FP16 的范围：     [-65504, 65504]  ← logits [200, 100, 50] 完全安全 ✅
FP16 的 exp 限制：exp(z) ≤ 65504  ← exp(11.09) ≈ 65504
                                     exp(11.1) → 溢出！

所以实际上：
  logits = [200, 100, 50]
  exp(200)  ≈ 7.2 × 10⁸⁶  →  远超 65504 → 溢出 ❌
  exp(100)  ≈ 2.7 × 10⁴³  →  远超 65504 → 溢出 ❌
  exp(50)   ≈ 5.2 × 10²¹  →  远超 65504 → 溢出 ❌
  exp(11.1) ≈ 66171       →  刚好溢出 ❌
```

问题不是 logits 存不下，而是 **softmax 计算中 exp(z_i) 超过了 FP16 的表示范围**。FP16 下只要 z > ~11 就会溢出，这个阈值比想象中小得多。

那 logsumexp trick 不就能解决吗？标准实现确实会做 max-subtraction：

```
logits = [200, 100, 50]
max_z  = 200

softmax = exp(z_i - 200) / Σexp(z_j - 200)
       = [exp(0), exp(-100), exp(-150)] / [exp(0) + exp(-100) + exp(-150)]

exp(0)    = 1           ✅
exp(-100) = 3.7 × 10⁻⁴⁴  → FP16 精度不足 → 下溢为 0
exp(-150) = 1.9 × 10⁻⁶⁵  → FP16 精度不足 → 下溢为 0
```

用 max trick 后，**溢出变成了下溢（underflow）**——非最大项变成 0，softmax 退化成 one-hot `[1, 0, 0]`。

Z-loss 真正要防止的是训练过程中 logits 增长到这个地步：
- **exp() 步骤溢出**（FP16 下 z > ~11 就溢出）
- 即使 max trick 把溢出变成下溢，**softmax 也会退化成 one-hot**，导致路由坍缩
- Z-loss 的核心目的是在训练过程中**防止 logits 增长到这个地步**，而不是事后补救

## 模块 3：MoE 在 LLM 中的实践 — QA 记录

**Q**：教程中 ByteTokenizer 的对比表提到 BPE 有 OOV 问题，但 BPE 从字节开始构建不是没有 OOV 问题吗？

**A**：

你说得对，这个表述不精确。

**BPE 实际上也没有 OOV 问题**——至少在现代实现中。关键在于"BPE 有两种退化策略"：

```
传统 BPE（字符级，Sennrich 2016 原始论文）:
  词表由训练语料的高频字符对合并而来
  如果某个 Unicode 字符在训练集中从未出现 → 退化为 <UNK>
  → 有 OOV（但极其罕见）

现代 BPE（字节级，GPT-2/3/4 使用的方案）:
  基础词表 = 256 个字节 → 任何 UTF-8 文本都能表示
  未见过的词 → 无法合并 → 退化为单字节序列
  → 无 OOV，只是 tokenization 更长
```

所以教程中"BPE 有 OOV 问题"这个说法不准确。**字节级 BPE 和字节级分词器都没有 OOV**。区别在于：

```
ByteTokenizer（本代码）:
  "cat" → [c, a, t] = 3 个 token（固定，无论训练数据是什么）
  "猫"  → [229, 165, 189] = 3 个 token（UTF-8 三字节）

BPE（训练后）:
  "cat" → [cat] = 1 个 token（高频词，合并为一个）
  "猫"  → [猫] = 1 个 token（如果训练数据有中文）
  或 [229, 165, 189] = 3 个 token（如果训练数据无中文，退化为字节）
```

真正的差异不是 OOV，而是**压缩率**：BPE 对高频词合并后序列更短（"cat" 只需 1 token），ByteTokenizer 永远按字节拆分（"cat" 始终 3 token）。已修正教程中的表述。

**Q**：教程中代码 `q = q.view(B, T, self.nhead, self.d_k).transpose(1, 2)`，transpose 不是不能保证连续吗，需要 contiguous 来让底层数据连续，为什么这里不需要 contiguous？

**A**：

好问题。这里的关键是**哪些操作需要 contiguous，哪些不需要**。

**`view` 需要 contiguous，`matmul` 不需要**：

```python
# 这里用了 view → 需要 contiguous 吗？
q = q.view(B, T, self.nhead, self.d_k)    # ✅ view 之前 q 是 contiguous 的
q = q.transpose(1, 2)                      # transpose 产生非连续视图，但这没问题
                                            # 因为后续没有再对 q 做 view

# matmul 不关心是否 contiguous——它按 stride 读数据，不假设内存连续
scores = torch.matmul(q, k.transpose(-2, -1))   # ✅ 非连续也能算

# 唯一需要 contiguous 的地方：
out = out.transpose(1, 2).contiguous().view(B, T, D)
                    # ↑ transpose 后要 view，所以必须先 contiguous
```

**`transpose` 做了什么？**

```
原始张量 shape=[B, T, H, d_k]，stride=[T*H*d_k, H*d_k, d_k, 1]
                                              ↓
transpose(1, 2) 只是交换 stride[1] 和 stride[2]，不移动任何数据
新 shape=[B, H, T, d_k]，stride=[T*H*d_k, d_k, H*d_k, 1]
                                              ↑
                     stride 不再是降序排列 → "非连续"
                     但数据在内存中完全一样，只是读取顺序变了
```

**为什么 `matmul` 能处理非连续输入？**

`matmul` 底层调用的是 BLAS/cuBLAS 矩阵乘法，它按 stride 逐元素读取，不要求数据在内存中连续排列。

**`view` 为什么必须 contiguous？**

`view` 的实现是：`shape × stride → 新 shape × 新 stride`，假设元素在内存中按行优先紧密排列。如果 stride 非降序（transpose 后的情况），view 无法推导出正确的新 stride，直接报错。

**总结：代码中 `contiguous()` 出现的位置完全对应 `view` 的调用**：

```python
# 第一处：reshape 进入多头
q = q.view(B, T, self.nhead, self.d_k).transpose(1, 2)
# ↑ view 在 transpose 之前，此时 q 刚从 qkv.chunk 出来，是 contiguous 的 ✅

# 第二处：reshape 回单头
out = out.transpose(1, 2).contiguous().view(B, T, D)
# ↑ view 在 transpose 之后，必须先 contiguous ✅
```

---

**Q**：为什么 `nn.Linear(d_model, vocab_size, bias=False)` 的 weight.shape 是 `[vocab_size, d_model]`，linear 内部结构是什么样的？weight 需要转置再和输入做乘积吗？

**A**：

`nn.Linear(in_features, out_features)` 的内部结构：

```
内部参数:
  weight: [out_features, in_features] = [vocab_size, d_model]
  bias:   无（如果 bias=False）

前向计算（PyTorch 默认公式）:
  y = x @ weight.T + bias
      ──   ────────
      │       │
      │       └── weight [vocab_size, d_model] → 转置 → [d_model, vocab_size]
      └── x [B, T, d_model]

  结果: [B, T, d_model] @ [d_model, vocab_size] = [B, T, vocab_size] ✅
```

**为什么 weight 存成 `[out, in]` 而不是 `[in, out]`？**

这是 PyTorch 的历史约定，公式写成 `y = xA^T + b`（数学文献中常见的矩阵乘法形式）。好处是**行向量 = 一个样本的权重**：

```
weight[vocab_size, d_model] 的第 i 行 = 第 i 个输出神经元的权重向量

weight[0, :] = [w0, w1, ..., w_{d_model-1}]  ← 输出位置 0 的权重
weight[1, :] = [w0, w1, ..., w_{d_model-1}]  ← 输出位置 1 的权重
```

`nn.Linear` 内部**确实做了转置**，但这是自动的（`x @ weight.T`），调用者不需要手动操作。`lm_head.weight` shape `[vocab_size, d_model]` 就是 `[out, in]` 的标准存储格式。
