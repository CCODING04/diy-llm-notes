## 核心内容

### 一、GPU 执行模型：Thread → Warp → Block

GPU 运行时，线程按三个粒度层级组织：

```
┌─────────────────────────────────────────────────┐
│  Block（线程块）                                  │
│  ┌───────────────────────────────────────────┐  │
│  │  Warp 0 (线程 0~31)    ← SM 调度最小单元   │  │
│  │  Warp 1 (线程 32~63)                      │  │
│  │  Warp 2 (线程 64~95)                      │  │
│  │  ...                                      │  │
│  │  Warp N                                    │  │
│  └───────────────────────────────────────────┘  │
│  共享内存（Block 内所有线程可见）                    │
└─────────────────────────────────────────────────┘
         ↓ 映射到
┌─────────────────────────────────────────────────┐
│  SM（流式多处理器）                                │
│  同时驻留多个 Block，管理数千线程                    │
└─────────────────────────────────────────────────┘
```

#### 1.1 Thread（线程）— 最细粒度

```
线程 = 流水线上的一个工人
  每个线程执行同样的 Kernel 代码，但操作不同数据
  每个线程有私有寄存器（通常 255 个）
  线程 ID（threadIdx.x）决定它处理哪个数据元素
```

#### 1.2 Warp（线程束）— 调度最小单元

```
Warp = 32 个线程组成的固定小组 = SM 调度的最小单元

类比：公交车
  32 个乘客（线程）必须同站同下
  → 执行完全相同的指令
  → 如果部分线程要走不同分支（if-else），全车人要等
  → 这就是 Warp Divergence（线程束分化）
```

**Warp 的硬件细节**（A100）：
- SM 同时驻留 **64 个 Warp**，最大 2048 线程/SM
- 4 个 Warp 调度器，每个可 dual-issue
- Warp 内 32 线程在 SIMD 单元上同步执行
- 执行上下文（PC、寄存器）在整个 Warp 生命周期内**常驻片上** → Warp 切换**零开销**
- SM **顺序发射指令**，**不做分支预测**，**不做推测执行**（与 CPU 不同）

> 🌐 **补充（Web Search）**：A100 支持 Compute Capability 7.0+ 的"独立线程调度"（Independent Thread Scheduling）——每个线程有独立的 PC 和调用栈，线程可以在 sub-warp 粒度上发散和重汇合，但需要显式 `__syncwarp()` 同步。来源：[NVIDIA CUDA Programming Guide](https://docs.nvidia.com/cuda/cuda-programming-guide/03-advanced/advanced-kernel-programming.html)

#### 1.3 Block（线程块）— 资源分配单位

```
Block = 程序员指定的线程组 → 映射到 1 个 SM

类比：施工队
  队内成员可以共享工具（共享内存）
  可以通过哨子同步（__syncthreads()）

资源独占：
  每个 Block 独占 SM 的共享内存和寄存器资源
  Block 内所有线程必须在同一 SM 内执行（不能跨 SM）
```

#### 1.4 SIMT（单指令多线程）

SIMT 是 GPU 的核心执行范式：

```
SIMT = Single Instruction Multiple Threads
  多个线程（Warp）共享同一条指令，但操作不同数据

类比：大合唱
  指挥（指令）统一
  每个人唱自己的声部（数据不同）

硬件实现：
  Warp 调度器发射 1 条指令 → 广播给 32 个 CUDA Core
  → 32 个 Core 同时执行相同指令，各自处理不同数据
```

### 二、Warp Divergence：SIMT 的代价

当 Warp 内线程遇到条件分支时：

```c
if (threadIdx.x < 16) {
    A;  // 线程 0~15 走这里
    B;
} else {
    X;  // 线程 16~31 走这里
    Y;
}
Z;
```

```
执行过程（硬件用掩码屏蔽实现串行化）：

步骤 1: 执行 if 分支
  线程 0~15:  ✅ 执行 A, B
  线程 16~31: ❌ 被屏蔽，空转等待

步骤 2: 执行 else 分支
  线程 0~15:  ❌ 被屏蔽，空转等待
  线程 16~31: ✅ 执行 X, Y

步骤 3: 继续执行 Z（全部线程）

总耗时 ≈ A+B 的时间 + X+Y 的时间（两条路径都走了）
```

**优化原则**：尽量避免同一线程束内的线程发散。如果一个 Warp 内所有线程都走同一路径，分支开销几乎为零。

### 三、TPU 架构：GPU 的"表亲"

TPU（Tensor Processing Unit）是 Google 自 2015 年起自研的 ASIC 芯片，专为神经网络计算而生。如果说 GPU 是"图形计算王者转职 AI"，TPU 就是"生来只为 AI"。

#### 3.1 TPU 与 GPU 的结构对比

```
GPU SM（流式多处理器）              TPU 张量核心
━━━━━━━━━━━━━━━━━━━━━              ━━━━━━━━━━━━━━━━━━━

┌──────────────────────┐          ┌──────────────────────┐
│  控制逻辑（分支判断）   │          │  标量单元（控制逻辑）   │
│  64 CUDA Core（通用）  │          │  矢量单元（逐项运算）   │
│  4 Tensor Core（矩阵） │          │  MXU（矩阵乘法单元）   │ ← 芯片最大面积
│  192KB 共享内存        │          │  矢量内存 + SM 内存    │
│  256KB 寄存器          │          │                       │
└──────────────────────┘          └──────────────────────┘
         ↕ 芯片外                          ↕ 芯片外
   HBM2e 全局内存                    HBM 全局内存
```

| 维度 | GPU | TPU |
|------|-----|-----|
| 核心理念 | 通用并行计算 | 专精矩阵乘法 |
| 计算单元 | CUDA Core（通用）+ Tensor Core（矩阵） | MXU（纯矩阵）+ 向量单元 + 标量单元 |
| 脉动阵列规模 | Tensor Core 32×32 | MXU 128×128（v2~v5）/ 256×256（v6e 起） |
| 每周期乘加次数 | 1,024（32×32） | 16,384（128×128）/ 65,536（256×256） |
| 执行模型 | Warp（32 线程 SIMT） | Block（无 Warp 概念） |
| 精度 | FP32/TF32/FP16/BF16/INT8 | BF16 输入，FP32 累加 |
| 可编程性 | 高（CUDA 生态） | 较低（JAX/TensorFlow） |
| 适用场景 | 通用 AI + 图形 + 科学计算 | 纯 AI 训练/推理 |
| 可扩展性 | 增加 SM 数量 | 增加张量核心数量，3D 环面互连（ICI） |

> 💡 **关键差异**：GPU 的 SM 尝试执行所有类型的计算（通用性），TPU 的 MXU 只做矩阵乘法（专用性）。TPU 架构更简洁，但灵活性不如 GPU。

> 🌐 **补充（Web Search）**：TPU 的 MXU 脉动阵列在矩阵乘法期间**无需访存**——数据在 MAC 单元网格中以波浪式流动，中间结果直接传递到下一个单元。v5p 和 TPU7x 还引入了 SparseCore（每芯片 4 个），加速稀疏操作。来源：[Google Cloud TPU 官方架构文档](https://docs.cloud.google.com/tpu/docs/system-architecture-tpu-vm)

### 四、性能扩展趋势与内存墙

#### 4.1 矩阵运算的"分道扬镳"

![矩阵运算速度](<../images/6-8-矩阵运算在各个设备的速度.png>)
*图6.8 矩阵运算在各设备的速度*

从 P100 开始，GPU 的非矩阵运算（蓝色）和矩阵运算（黄色）出现巨大差异——这正是 Tensor Core 的功劳。Tensor Core 是专门为矩阵乘法设计的电路，不是缩小版 CUDA Core。

#### 4.2 计算与内存扩展的不平衡

![计算与内存扩展](<../images/6-9-计算速度扩展的比内存扩展速度更快.png>)
*图6.9 计算速度扩展比内存扩展速度更快*

```
1980~2000：登纳德缩放定律
  晶体管缩小 → 频率提升 → 功耗下降
  单线程性能持续提升

2000 年后：登纳德缩放失效
  单线程性能趋于平缓，无法靠频率提升
  现代扩展方式：并行扩展（增加 SM 数量）

结果（K20 → H100）：
  计算性能: ~10 万倍提升  ← 最快
  内存带宽: ~100 倍提升   ← 慢得多
  互联带宽: 增长最缓慢    ← 最慢

核心矛盾：计算能力增长远快于内存带宽
  → "内存墙"问题
```

#### 4.3 屋顶线模型（Roofline Model）

屋顶线模型是分析程序性能瓶颈的核心工具，由 Samuel Williams、Andrew Waterman、David Patterson 于 2008 年提出。

![屋顶线模型](<../images/6-10-屋顶线模型.png>)
*图6.10 屋顶线模型*

```
横轴: 操作强度 / 算术强度（Operational Intensity）
      = 计算量 / 数据移动量（FLOPs / Byte）
      → 每搬运 1 字节数据，能做多少次计算

纵轴: 吞吐量（Throughput）
      = 每秒完成的浮点运算次数（FLOPS）

两个硬件上限（"屋顶"）：
  计算屋顶（水平线）= 峰值算力（FLOPS）
  内存屋顶（斜线）  = 内存带宽（Bytes/sec）× 操作强度

脊点（Ridge Point）= 两条线交点的 X 坐标
  → 脱离内存瓶颈所需的最小操作强度
  → 随着计算增长远超内存带宽增长，脊点不断右移
```

> 🌐 **补充（Web Search）**：内存墙概念由 Wulf & McKee 于 1994 年提出。Patterson 于 2004 年进一步指出"延迟的线性改善伴随着带宽的二次改善"。LLM 推理的 Decode 阶段算术强度极低（约 1 FLOP/byte），严重 memory-bound，是当前推理优化的核心挑战。来源：[Modal GPU Glossary](https://modal.com/gpu-glossary/perf/roofline-model)

**屋顶线模型的两个区域**：

```
                    ╱ GPU ALU throughput（算力天花板）
                   ╱
                  ╱
                 ╱
                ╱
               ╱  ← 拐点（ridge point）
              ╱
             ╱
    ╱───────╱
   ╱ 内存受限区
  ╱  （Memory Bound）
 ╱
────────────────────────────────→ 操作强度

左侧区域 = 内存受限（Memory Bound）
  数据搬运太慢，计算单元空转等待数据
  → 瓶颈在内存带宽
  → 优化方向：减少数据搬运（Tiling、算子融合）

右侧区域 = 计算受限（Compute Bound）
  计算单元满载，数据供应充足
  → 瓶颈在算力
  → 优化方向：用更快的精度（FP16/INT8）或更多 Tensor Core
```

#### 4.4 内存墙的启示

> 💡 **核心洞察**：算法设计必须**以内存为中心**。计算能力增长 10 万倍，内存带宽只增长 100 倍——差距是 1000 倍。这意味着如果不精心管理数据搬运，GPU 的算力会被严重浪费。

```
AI 训练的现实：
  计算: Tensor Core 飞速运转
  等待: 数据从 HBM 搬运到 SM

  → 后续模块的所有优化技术（Tiling、算子融合、重计算、
    FlashAttention）本质上都是在解决这个内存墙问题
  → 目标：让计算单元尽可能少等数据
```
