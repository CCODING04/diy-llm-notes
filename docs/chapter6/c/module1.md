## 核心内容

### 一、GPU 的起源：从图形处理器到 AI 引擎

GPU 最初是**图形处理器**——游戏显卡。3D 模型由大量小三角形构成，每个三角形只存储三个顶点坐标，中间像素需要实时计算渲染。这些计算本质是**大量简单乘法和加法**，CPU 虽然能做，但它是为复杂逻辑优化的，只能逐个计算，速度极慢。

于是人们创造了 GPU——一种**大量并行计算简单乘法和加法**的计算单元。到了 AI 时代，深度学习的核心是**矩阵运算**，本质也是大量乘法和加法，GPU 从"游戏显卡"升级为"AI 发动机"。

### 二、CPU vs GPU：设计哲学的根本差异

CPU 和 GPU 没有优劣之分，设计目标截然不同：

```
CPU（中心处理器）                    GPU（图形处理器）
━━━━━━━━━━━━━━━━━━━━━━━            ━━━━━━━━━━━━━━━━━━━━━━━

优化目标:    单任务延迟最小化           吞吐量最大化
设计哲学:    单个任务最快完成           所有任务整体最快完成

芯片面积分配:
┌──────────────────────┐            ┌──────────────────────┐
│ ████ Control ████████ │            │ ▪ Control ▪          │
│ ████ Cache   ████████ │            │                       │
│ ████ Branch  ████████ │            │ ████████████████████ │
│ ██ Pred.    ████████ │            │ ████████████████████ │
│     ┌──┐             │            │ ██████ ALU █████████ │
│     │AL│             │            │ ████████████████████ │
│     └──┘             │            │ ████████████████████ │
└──────────────────────┘            └──────────────────────┘

核心数:     4~64 核                   数千~数万 CUDA 核心
频率:       极高（~5GHz）              较低（~1.5GHz）
擅长:       复杂逻辑、分支预测          大量并行的简单乘加
```

| 特性 | CPU | GPU |
|------|-----|-----|
| 晶体管用途 | 控制逻辑 + 缓存（大） | ALU 计算单元（大） |
| 核心数量 | 4~64 | 数千~数万 |
| 单线程能力 | 极强（乱序执行、分支预测） | 弱 |
| 并行能力 | 弱 | 极强 |
| 线程切换开销 | 高（保存大量状态） | 极低（轻量级切换） |
| 优化目标 | **延迟**（单任务最快） | **吞吐量**（整体最快） |

> 💡 **关键认知**：GPU 不是"更好的 CPU"，而是完全不同的计算范式。GPU 需要 CPU 来调度——一个好的 GPU 要配上好的 CPU 才能发挥作用。

### 三、A100 GPU 的层级架构

A100 是 NVIDIA 为数据中心设计的纯计算 GPU（无图形输出），采用 Ampere 架构。

#### 3.1 五层架构拓扑

```
第五层：A100 整卡
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  供电 + GPU 核心 + HBM2e 显存（High Bandwidth Memory，高带宽内存）+ 金手指
  542 亿晶体管，7nm 工艺，面积 826mm²

第四层：GA100 GPU 核心
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ┌─────────┐ ┌─────────┐ ┌─────────┐
  │ GPC 0   │ │ GPC 1   │ │ GPC 2   │     ← 8 个 GPC
  │(12 TPC) │ │(12 TPC) │ │(12 TPC) │
  └─────────┘ └─────────┘ └─────────┘
  ┌─────────┐ ┌─────────┐ ┌─────────┐
  │ GPC 3   │ │ GPC 4   │ │ GPC 5   │
  └─────────┘ └─────────┘ └─────────┘
  ┌─────────┐ ┌─────────┐ ┌─────────┐
  │ GPC 6   │ │ GPC 7   │ │ GPC 8   │
  └─────────┘ └─────────┘ └─────────┘

第三层：GPC（图形处理簇）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  每 GPC = 12 TPC
  共 96 TPC

第二层：TPC（纹理处理簇）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  每 TPC = 2 SM
  共 192 SM（实际启用 108 个）

第一层：SM（流式多处理器）← 核心执行单元
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  每 SM 包含：
  ├── 64 个 CUDA Core（FP32/INT32 计算单元）
  ├── 4 个 Tensor Core（第 3 代，AI 专用加速）
  ├── 192KB 共享内存 / L1 缓存
  └── 256KB 寄存器文件
```

#### 3.2 关键数量汇总

| 组件 | 数量 | 说明 |
|------|------|------|
| GPC | 8 个 | 图形处理簇 |
| TPC | 96 个（8×12） | 纹理处理簇 |
| SM | 108 个（启用） | 流式多处理器，**核心执行单元** |
| CUDA Core | 6,912 个（108×64） | FP32/INT32 通用计算 |
| Tensor Core | 432 个（108×4） | 矩阵乘法专用加速 |

#### 3.3 Tensor Core：AI 专用加速单元

Tensor Core 不是缩小版 CUDA Core，而是**重构的矩阵引擎**，采用 **脉动阵列（Systolic Array）** 架构：

```
数据流:  A[i,k] ──→ ┌──────────┐
                    │ 脉动阵列  │ ──→ C[i,j]（累加结果）
         B[k,j] ──→ │ 32×32    │
                    └──────────┘
每个周期每个单元完成 1 次乘加
32×32 阵列每周期 = 1024 次运算
```

A100 Tensor Core 支持的精度与性能：

| 数据类型 | 峰值算力（每 SM） | 相对 FP32 CUDA Core |
|----------|-------------------|---------------------|
| FP64 | 19.5 TFLOPS | 1×（基准） |
| TF32 | 156 TFLOPS | **8×** |
| FP16/BF16 | 312 TFLOPS | **16×** |
| INT8 | 624 TOPS | **32×** |
| INT4 | 1,248 TOPS | **64×** |

> 💡 **补充（Web Search）**：FlashAttention 在 A100 上配合优化的 MLP/LayerNorm/CE loss，比 HuggingFace 基线快 **3-5x**，达到 **225 TFLOPs/sec**（72% 模型 FLOPs 利用率）。来源：[Dao-AILab/flash-attention](https://github.com/Dao-AILab/flash-attention)

**核心技术**：
- **结构化稀疏**：自动跳过 0 值计算，有效性能翻倍
- **多精度融合**：单周期内处理不同精度数据
- **权重静止**：矩阵权重预加载到阵列寄存器，减少数据搬运
- **FP32 累加器**：输入输出用低精度，累加保持高精度，保证训练稳定性

### 四、GPU 内存层次结构

这是理解 GPU 优化的**核心基础**——所有 FlashAttention、Tiling、算子融合等优化技术，本质上都是在不同内存层次之间减少数据搬运。

#### 4.1 四层内存总览

```
┌─────────────────────────────────────────────┐
│                 SM 内部                       │
│  ┌───────────────────────────────────────┐  │
│  │  寄存器文件（Register File）            │  │
│  │  256KB/SM | 1 周期 | 线程私有           │  │
│  └───────────────────────────────────────┘  │
│  ┌───────────────────────────────────────┐  │
│  │  共享内存（Shared Memory）/ L1 缓存     │  │
│  │  192KB/SM | 20-40 周期 | Block 内共享   │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
                    ↕ 芯片内
┌─────────────────────────────────────────────┐
│  L2 缓存 — 40MB | ~200 周期 | 全 GPU 共享    │
└─────────────────────────────────────────────┘
                    ↕ 芯片外（HBM 连接器）
┌─────────────────────────────────────────────┐
│  全局内存（HBM2e）— 80GB | ~500 周期          │
└─────────────────────────────────────────────┘
```

#### 4.2 各层详细参数

| 内存类型 | 物理位置 | 容量 | 带宽 | 延迟 | 编程控制 | 可见性 |
|----------|---------|------|------|------|---------|--------|
| **寄存器** | SM 内，CUDA Core 旁 | 256KB/SM | 10TB/SM | **1 周期** | 自动（编译器） | 线程私有 |
| **共享内存** | SM 内 | 192KB/SM | 1TB/SM（**SM 独占**） | **20-40 周期** | 手动（`__shared__`） | Block 内共享 |
| **L2 缓存** | 芯片内，全 GPU 共享 | 40MB | ~3TB/s（108 SM 共享） | **~200 周期** | 自动（硬件） | 全 GPU |
| **全局内存** | 芯片外 HBM2e | 80GB | 2,039 GB/s（108 SM 共享） | **~500 周期** | 手动（`cudaMalloc`） | 全 GPU |
| **常量内存** | 芯片内专用缓存 | 64KB | 广播机制 | ~5 周期（命中） | 只读（`__constant__`） | 全 GPU 只读 |

#### 4.3 关键延迟对比

```
寄存器:  █ 1 周期
共享内存: ████████████████████ 20-40 周期
L2 缓存: ██████████████████████████████████████████████ 200 周期
全局内存: █████████████████████████████████████████████████████████████████████████ 500 周期

总带宽差距:
  共享内存 vs 全局内存 → 10~25 倍延迟
  寄存器 vs 全局内存 → 500 倍延迟

每 SM 实际可用带宽（关键！）:
  共享内存: 1 TB/s（SM 独占，不用分）
  L2 缓存:  ~28 GB/s（3 TB/s ÷ 108 SM）
  全局内存: ~18.5 GB/s（2 TB/s ÷ 108 SM）

  共享内存 vs 全局内存 → 实际 ~54 倍带宽差距
  L2 vs 全局内存 → 实际仅 ~1.5 倍（差距不大，L2 的价值在缓存命中）
```

> ⚠️ **核心认知**：距离 SM 越近的内存，访问速度越快。**GPU 性能优化的本质就是尽可能让数据待在快速内存中**，减少对慢速全局内存的访问。

#### 4.4 各层内存的设计目的

```
全局内存（HBM2e）：
  → 海量容量（80GB），存放模型权重、激活值、梯度
  → 成本最低（相对 SRAM 即静态随机存取存储器），但速度最慢
  → CPU-GPU 数据传输通道（PCIe，Peripheral Component Interconnect Express）

L2 缓存：
  → 全局数据的"加速器"，自动缓存热点数据
  → SM 间数据共享枢纽（比走 HBM 快 3 倍）
  → 硬件自动管理，程序员无需干预

共享内存：
  → Block 内线程协作的"仓库"
  → 程序员手动控制，可实现接近寄存器的速度
  → GPU 性能优化的**核心战场**
  → 未手动使用时，自动作为 L1 缓存

寄存器：
  → 零延迟计算，存储线程局部变量和临时结果
  → 容量限制决定并行度：寄存器用量越少，SM 能驻留的 Warp 越多
  → 速度快但昂贵，容量有限

常量内存：
  → 广播机制：同一 Warp 内 32 线程读同一常量，只需 1 次访问
  → 存储学习率、查找表等配置参数
```

### 五、GPU 内存分层的根本原因

#### 5.1 速度-容量-成本的三角权衡

| 内存类型 | 速度 | 容量 | 成本（估计 $/GB） |
|----------|------|------|------------------|
| 寄存器 | 1 周期 | 256KB/SM | ~$1,000,000 |
| 共享内存 | 20 周期 | 192KB/SM | ~$100,000 |
| L2 缓存 | 200 周期 | 40MB | ~$10,000 |
| HBM2e 显存 | 500 周期 | 80GB | ~$100 |

```
如果只用寄存器：80GB ÷ 256KB = 327,680 个 SM 的寄存器才够
               → 成本天文数字，不可能

如果只用显存：算力浪费 90%（等待数据搬运）
               → GPU 利用率极低

分层是唯一经济可行方案：用少量昂贵的快速内存 + 大量便宜的慢速内存
```

#### 5.2 局部性原理驱动分层设计

```
时间局部性：刚访问的数据很可能再次访问
  → L2 缓存利用此特性，缓存重复访问的权重

空间局部性：相邻数据很可能一起访问（如矩阵同行元素）
  → 共享内存利用此特性，手动加载分块（Tiling）数据

广播特性：同一 Warp 的 32 个线程可能读同一常量
  → 常量内存 1 次读取服务 32 线程
```

#### 5.3 GPU vs CPU 内存的本质区别

| 特性 | GPU (A100) | CPU (Xeon) |
|------|------------|------------|
| 主存带宽 | 2 TB/s | 100 GB/s |
| 缓存控制 | 共享内存**手动控制** | 缓存完全自动 |
| 线程寄存器 | 255 个/线程 | 16 个/线程 (x86) |
| 延迟容忍 | 通过 Warp 切换**隐藏延迟** | 降低延迟至上 |
| 内存模型 | 共享内存显式同步 | 缓存一致性协议 |

> 💡 **核心洞察**：GPU 内存系统**为吞吐量优化**，容忍高延迟（通过大量线程切换隐藏）；CPU 内存系统**为延迟优化**，追求最低延迟。GPU 需要更多层级和手动控制，正是因为它的设计目标不同。

**A1**：
A100 有 8 个 GPC
每个 GPC 有 12 个 TPC
每个 TPC 有 2 个 SM
每个 SM 有 64 个 CUDA core、4 个 Tensor Core

SM 是流式处理器，包括完整的寄存器、存储器 和 ALU（CUDA core） 部分，是核心计算单元。


**A2**：
GPU 的四层内存执行周期分别为
- 寄存器： 1 周期
- 共享内存： 20 周期
- L2：200 周期左右
- 全局内存： 500 周期


**A3**：

GPU 之所以使用分层架构，主要是 速度、成本的 均衡结果。
80GB 的存储空间如果都使用高速内存，成本无法控制，会导致无人可买；而如果都使用 HBM，则 速度很慢，计算速度很快 IO 速度很慢，时间都浪费在  IO 上了，无法发挥计算的优势。
而使用多层架构，既能让 靠近 CUDA core 的部分使用高速内存，提高数据搬运的速度和计算匹配，在全局内存中使用 HBM，又能保证存储空间，并且在中间添加 L2 cache 和 共享内存，提高数据搬运的速度和缓存大小，让数据逐级 IO，既能保证存储空间又能稳定搬运的速度。
